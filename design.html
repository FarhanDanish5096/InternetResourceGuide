Characteristics: Design

Design

Git’s design was shaped by Linus Torvalds’s experience with Linux kernel development, prioritizing performance, flexibility, and distributed collaboration.

Key Design Principles

  • Non-linear development support: Git makes branching and merging easy and lightweight, enabling multiple concurrent development streams.
  • Distributed architecture: Each developer has a complete local history of the project, allowing offline work and flexible collaboration.
  • Compatibility: Supports multiple protocols (HTTPS, SSH, FTP), and works with older systems like CVS and Subversion.
  • Performance: Optimized for speed and scalability, especially with large codebases.
  • Security: Uses cryptographic hashing (SHA-1) to ensure the integrity of the repository history.
  • Modular and scriptable: Initially designed as a toolkit of small programs, allowing composability and extensibility.
  • Efficient storage: Uses snapshot-based storage (not per-file history), with loose objects and compressed packfiles.
  • Garbage collection and packing: Unreferenced objects are cleaned up over time using git gc; storage is compacted via periodic repacking.
  • Snapshot model: Git tracks the state of entire directory trees, not individual file histories.
  • File rename detection: Instead of explicitly tracking renames, Git infers them during history examination, which can be CPU-intensive.
